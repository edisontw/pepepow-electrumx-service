Gateway Safety Boundary

Security Notice: PEPEW Light Wallet is non-custodial. Mnemonic phrases and private keys stay in the browser. The PEPEW Light API only receives addresses, transaction IDs, and future signed raw transactions.

Read-Only Architecture

We do not store private keys, seed phrases, or wallet configuration. Safe from credential leaks.

No Custody or Ledgers

There is no invoice database. The monitor checks real-time address balances. Generate a fresh address per payment for safety.

Private Node Access

Our ElectrumX server is isolated and bound only to the local interface, routing all traffic through rate limiters.

Public API Documentation

Developers can query the API using the following REST endpoints:

GET /api/health
Gateway heartbeat

Purpose

Lightweight ping to verify the API Gateway is online.

Example request

curl -s https://light.pepepow.net/api/health
GET /api/status
ElectrumX node state & sync height

Purpose

Reports upstream connection status, block height, and protocol info.

Example request

curl -s https://light.pepepow.net/api/status
GET /api/address/{address}
Address balance & metadata

Purpose

Inspects confirmed/unconfirmed balances (in satoshis) and transaction counts.

Example request

curl -s https://light.pepepow.net/api/address/PRfbEeHAKKbz6Voz85WJudrJwTA3ZbHunb
GET /api/address/{address}/history
Paginated history lookup

Purpose

Paginates through address transaction hashes.

Example request

curl -s "https://light.pepepow.net/api/address/PRfbEeHAKKbz6Voz85WJudrJwTA3ZbHunb/history?limit=5&offset=0"
GET /api/tx/{txid}
Transaction lookup

Purpose

Retrieve transaction block height, confirmation count, input/output data, and full hex values.

Example request

curl -s https://light.pepepow.net/api/tx/453c4f723b2eb5cf4b8ef9818fac50d08ddc8ed05bc972ee464f2dbc55b6a3db
GET /api/payment/check
Address-level payment verifier

Purpose

Evaluates if the current visible balance of an address meets the requested amount.

Example request

curl -s "https://light.pepepow.net/api/payment/check?address=PRfbEeHAKKbz6Voz85WJudrJwTA3ZbHunb&amount=1"

Response fields (Preferred balance fields)

Field Type Description
statusstringPayment status: waiting, seen_in_mempool, partial, paid_unconfirmed, paid_confirmed, overpaid, or expired.
confirmed_balancestringConfirmed address balance in PEPEW.
confirmed_balance_satsintegerConfirmed address balance in satoshis.
mempool_balancestringUnconfirmed mempool balance in PEPEW.
mempool_balance_satsintegerUnconfirmed mempool balance in satoshis.
total_visible_balancestringTotal visible balance in PEPEW (confirmed + mempool).
total_visible_balance_satsintegerTotal visible balance in satoshis (confirmed + mempool).
requested_amountstringRequested amount in PEPEW.
requested_satsintegerRequested amount in satoshis.
expiredbooleanTrue if current server time exceeds the expiry limits.

Compatibility aliases (Deprecated)

The following fields are kept for backward-compatibility only and should not be used in new integrations:

confirmed_received, confirmed_received_sats, mempool_received, mempool_received_sats, total_received, total_received_sats, received_confirmed_sats, received_unconfirmed_sats

Safety note

This is not an invoice database. It checks address balances. For reliable checking, use a unique, unused receiving address for each payment request. Reusing addresses or spending from the address can make checks ambiguous.
